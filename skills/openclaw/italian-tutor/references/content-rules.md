# Content Rules

## Core Principle

Teach reusable chunks and sentence frames before abstract explanations.

## Preferred Content Shape

Prefer:

- short chunks
- high-frequency practical phrases
- pattern-friendly examples
- mission-relevant language
- recovery cards that encode the next correct move

Avoid:

- isolated noun dumps
- low-yield niche vocabulary
- awkward or overly literal sentences
- advanced grammar lists without communicative value
- artifact-like mistake traces stored as learner cards

## Learning Design

Optimize for:

- retrieval practice
- chunk repetition
- interleaving
- error-driven repair
- mission-linked production

Use explanations only to unlock use, not as the main product.

## Level Guidance

- `A1`: survival chunks, requests, location, basic repair, core routines
- `A2`: daily-life control, past events, planning, preferences, practical independence
- `B1`: narration, explanation, opinions, more flexible mission language
- `B2`: chat can stretch here, but runtime inventory should only be expanded deliberately

## Pattern Guidance

When generating pattern-oriented content, use supported runtime pattern IDs and lanes.

Examples of high-yield patterns:

- `polite_request_vorrei`
- `need_ho_bisogno_di`
- `movement_vado`
- `location_essere`
- `past_ho_participio`
- `plan_penso_di`
- `opinion_secondo_me`
- `conversation_repair`

## Recovery Rules

Recovery content should be:

- short
- reusable
- corrected
- easy to re-say
- tied to the learner's actual failure mode

Each item should answer:

- what should the learner say instead
- what pattern or chunk does this belong to

## Mission Rules

Mission content should:

- match the active mission scenario
- support the current checkpoint or bottleneck
- stay within the learner's usable level while nudging production upward

Do not write content that sounds like a textbook when the mission is practical and scenario-based.
